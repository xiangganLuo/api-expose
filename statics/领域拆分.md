### 1. 战略设计：限界上下文划分

基于系统架构的从底向上分层，我们可以划分为以下几个核心领域：

| 领域分类                    | 限界上下文 (Bounded Context)         | 核心职责                                                     | 业务层级映射                       |
| :-------------------------- | :----------------------------------- | :----------------------------------------------------------- | :--------------------------------- |
| **核心域** (Core Domain)    | **API 资产域 (Api Asset Context)**   | 负责API的解析、建模、生命周期管理（发布/下架）及版本控制。   | L2 开放平台核心层、L0 能力接入层   |
| **核心域** (Core Domain)    | **应用与订阅域 (App & Sub Context)** | 负责外部开发者的应用创建、鉴权凭证生成，以及应用对API的订阅申请与审批。 | L3 开发者体验层、L2 开放平台核心层 |
| **核心域** (Core Domain)    | **策略与治理域 (Policy Context)**    | 负责管理 API 和应用的访问控制、流量管控（限流/熔断）以及安全策略。 | L2 开放平台核心层                  |
| **支撑域** (Support Domain) | **计量计费域 (Metering Context)**    | 收集全网关调用流水，进行按应用/API/时间跨度的聚合统计、出账和套餐结算。 | L4 运营与管理层                    |
| **支撑域** (Support Domain) | **网关路由域 (Gateway Context)**     | 根据发布的 API 资产和策略，动态生成路由表，执行请求转发和拦截。 | L1 基础设施与引擎层                |
| **通用域** (Generic Domain) | **租户域 (Tenant Context)**          | 负责多租户的数据隔离、租户环境设置、成员与角色管理权限体系。 | L4 运营与管理层                    |
| **通用域** (Generic Domain) | **认证域 (Auth Context)**          | 负责开发者与管理员的统一身份认证、Session 管理、密码安全策略。 | L1 基础设施与引擎层                |

### 2. 战术设计：各域的核心模型

#### 2.1 API 资产域 (Api Asset Domain)

**语境**：负责管理被开放出去的能力，从外部导入 Swagger 等，到形成平台内的标准 API 定义。

- 聚合根 (Aggregate Root): ApiAsset(API资产)
  - **业务行为**: `importDefinition()`, `publish()`, `offline()`, `upgradeVersion()`
- 实体 (Entity):
  - `ApiEndpoint` (API端点，具体的路由路径及Method)
  - `ApiVersion` (API版本，支持多版本演进)
- 值对象 (Value Object):
  - `ApiDefinitionVO` (内部标准API解析模型：请求参数、响应结构树等)
  - `ApiStatusEnum` (状态：DRAFT, PUBLISHED, DEPRECATED, OFFLINE)
  - `ProtocolTypeEnum` (HTTP, HTTPS 等)

#### 2.2 应用与订阅域 (App & Subscription Domain)

**语境**：外部开发者作为消费者，在门户注册应用并订阅平台开放的 API，以此换取调用凭据（AK/SK）。

- 聚合根 1: DeveloperApp(开发者应用)

  - **业务行为**: `generateCredential()`, `resetSecret()`, `applySubscription()`
  - **实体**: `AppCredential` (应用安全凭证，如 API Key, Secret)
  - **值对象**: `AppStatusEnum` (启用/禁用状态)

- 聚合根 2: Subscription

  (订阅关系)

  - **业务行为**: `approve()`, `reject()`, `revoke()`
  - **值对象**: `SubscriptionStatusEnum` (PENDING, APPROVED, REJECTED, REVOKED)
  - **实体映射**: 记录 `AppId` 和对应订阅的 `ApiAssetId`

#### 2.3 策略与治理域 (Policy Governance Domain)

**语境**：保护内部系统的稳定，通过策略的形式灵活绑定到具体的 API 或具体的调用方应用上。

- 聚合根: GovernancePolicy(治理策略)
  - **业务行为**: `attachToApi()`, `attachToApp()`, `modifyRules()`
- 实体 (Entity):
  - `RateLimitRule` (限流规则：令牌桶大小，QPS阈值)
  - `CircuitBreakerRule` (熔断规则：错误率阈值，恢复时间窗口)
  - `AccessControlRule` (访问控制：IP白名单，OAuth2/AK模式开关)
- 值对象 (Value Object):
  - `PolicyScopeEnum` (生效范围：GLOBAL, API_LEVEL, APP_LEVEL)

#### 2.4 网关路由域 (Gateway Domain)

**语境**：运行时态的核心模型，将策略与资产转换为网关引擎可直接执行的运行时规则。

- 聚合根: RouteRule

  (路由规则)

  - **值对象**: `UpstreamTargetVO` (上游目标真实地址), `PathMappingVO` (网关路径映射逻辑)

### 3. 上下文映射关系 (Context Map)

通过 DDD 上下文映射，我们描述各领域之间的依赖及消息传递形式：

1. 策略域 / 订阅域 → 网关路由域 

   （基于最终一致性的事件流 / 同步分发规则）

   - *说明*：当 API 发布、限流策略修改、或应用订阅被通过时，系统会产生**领域事件**送达。网关域防腐并在内存/Redis中更新路由或限流标识。

2. API 资产域 ← 订阅域

   （防腐层 ACL 弱关联）

   - *说明*：订阅域不能直接持有复杂的 API 资产对象模型，只需存储值对象 `ApiAssetId` 和 `ApiVersion` 等标识。

3. 网关路由域 $\rightarrow$ 计量计费域

   （Customer-Supplier，MQ 流转）

   - *说明*：网关产生的海量调用流水投递到 MQ，计量域作为消费者异步聚合分析，不影响业务网关低延迟运行。
